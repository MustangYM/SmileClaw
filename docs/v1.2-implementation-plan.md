# SmileClaw v1.2 Implementation Plan

## Scope
This plan executes `SmileClaw_Architecture_Optimization_Document_v1.2` in incremental phases.
Target: move current prototype to a stable runtime baseline with policy, approvals, events, and metrics.

## Delivery Rules
- Keep runtime behavior backward-compatible where possible.
- Every phase must include tests or reproducible validation commands.
- No phase is complete until DoD checks pass.

## Phase 0 - Baseline and Safety Net
### Tasks
- [ ] P0-1 Add regression test harness for CLI agent loop and shell tool calls.
- [ ] P0-2 Add deterministic fixtures for approval scenarios (`allow_once`, `allow_always`, `deny`).
- [ ] P0-3 Add a `make check`/single command workflow to run lint + tests.

### DoD
- [ ] Existing approval flow still works.
- [ ] Reproducible local test command validates baseline.

## Phase 1 - Runtime Stabilization (v1.2 core)
### Track A - Shell Policy Evaluation Pipeline
- [ ] P1-A1 Add `security/policy.py` with pipeline stages:
  - parse
  - normalize
  - semantic extraction
  - policy evaluation
  - post-execution analysis
- [ ] P1-A2 Detect risky shell features:
  - separators (`;`, `&&`, `||`)
  - pipes
  - redirections (`>`, `>>`)
  - subshells (`$()`, backticks)
  - cwd traversal (`cd ..`, relative escapes)
- [ ] P1-A3 Produce structured policy result:
  - `decision`: `allow` | `require_approval` | `deny`
  - `restricted_paths`
  - `risk_flags`
  - `reason_codes`

### Track B - Approval Persistence and Priority
- [ ] P1-B1 Migrate `.smileclaw/exec-approvals.json` to v1.2 schema:
  - `version`
  - `rules[]`
- [ ] P1-B2 Implement rule priority exactly:
  - explicit deny
  - exact command allow
  - path root allow
  - session allow
  - default policy
- [ ] P1-B3 Support scope types:
  - `tool + path_root`
  - `tool + command_hash`
- [ ] P1-B4 Add audit events for:
  - rule created
  - rule revoked
  - rule expired

### Track C - Runtime Guard and Tool Runtime
- [ ] P1-C1 Add shell timeout support and return `TIMEOUT` error code.
- [ ] P1-C2 Add output size truncation with `output_truncated` flag.
- [ ] P1-C3 Add max command length check before execution.
- [ ] P1-C4 Enforce argument schema validation before tool invocation.

### Track D - Loop and Denial Safety
- [ ] P1-D1 Keep per-turn denied-command memory.
- [ ] P1-D2 Add bounded-step guard in runtime loop.
- [ ] P1-D3 Emit structured error on loop protection trigger.

### DoD
- [ ] Reject loops: same denied command is not re-approved in one turn.
- [ ] `cd .. && pwd` outside workspace always triggers approval.
- [ ] Timeout and output truncation are observable in tool result payload.
- [ ] Approval rules persist and are reloaded correctly across restarts.

## Phase 2 - Observability
### Track A - Event Protocol
- [ ] P2-A1 Add runtime event envelope:
  - `id`
  - `type`
  - `timestamp` (ISO8601)
  - `run_id`
  - `correlation_id`
  - `payload`
  - `error_code`
- [ ] P2-A2 Add minimal payload contract per event type.
- [ ] P2-A3 Guarantee strict ordering within same `run_id`.
- [ ] P2-A4 Add event sink abstraction (stdout/jsonl/file).

### Track B - Metrics Collection
- [ ] P2-B1 Add counters for:
  - policy decisions
  - approvals requested/resolved
  - tool execution success/failure/timeout
- [ ] P2-B2 Add rolling-window aggregator (7-day metrics).
- [ ] P2-B3 Report v1.2 metrics:
  - unauthorized execution rate
  - approval bypass rate
  - crash rate
  - timeout rate

### DoD
- [ ] Every tool call and approval emits events.
- [ ] Metrics report is generated from event/log data, not manual counting.

## Phase 3 - Context Governance and Prompt Builder
### Tasks
- [ ] P3-1 Upgrade memory record schema:
  - `role`
  - `content`
  - `name`
  - `metadata`
  - `timestamp`
- [ ] P3-2 Add token estimator and thresholds:
  - summarize when context exceeds 8k tokens
  - hard cap at 16k
- [ ] P3-3 Add rolling summary strategy preserving instruction priority:
  - `system > user > tool > assistant`
- [ ] P3-4 Introduce `PromptBuilder.build()` composition flow.

### DoD
- [ ] Long sessions do not exceed context cap.
- [ ] Critical user instructions survive summarization.

## Phase 4 - Extensibility
### Tasks
- [ ] P4-1 Refactor tool registration to `ToolDefinition` model.
- [ ] P4-2 Introduce plugin loader after contracts stabilize.
- [ ] P4-3 Add skills framework integration points.

### DoD
- [ ] New tool plugin can be added without modifying core runtime loop.

4. P2-A
5. P2-B
6. P3
